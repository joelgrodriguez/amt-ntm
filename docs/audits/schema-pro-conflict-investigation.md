# Schema Pro conflict investigation

- **Status:** Read-only investigation
- **Production captured:** 2026-07-18 (America/Phoenix)
- **Reported findings:** 2026-07-16
- **URLs:** [Homepage](https://newtechmachinery.com/) and [SSQ3 product page](https://newtechmachinery.com/machines/roof-wall-panel-machines/ssq3-multipro/)

## 1. Executive summary

Thirteen of the fourteen reported entity-level homepage errors still reproduce in current production output:

- Six homepage Product nodes have none of offers, review, or aggregateRating.
- Six homepage Review nodes have no itemReviewed. They also have no reviewRating, another Google review-snippet requirement.
- One homepage VideoObject has no uploadDate.

The reported homepage BreadcrumbList “missing item” error is not an active defect. Yoast omits item only from the current/last breadcrumb, which Google explicitly permits. Schema Pro emits a second BreadcrumbList with the same @id, however, so the real breadcrumb problem is duplicate and conflicting ownership.

All thirteen reproduced errors come from one Schema Pro custom-markup payload. In the local WordPress database, it is schema post **18578, “Home Page Custom Schema,”** targeted to special-front. After replacing the local hostname with the production hostname, its stored JSON is structurally and textually identical to production JSON-LD script 2. Schema Pro also caches that payload on the homepage in post meta key wp_schema_pro_optimized_structured_data, and its renderer returns that cache before rebuilding the rule.

The SSQ3 Product is not generated by Schema Pro. It comes from the theme’s machine-schema renderer and has a real AggregateOffer backed by the machine data file. Schema Pro reaches the page through its generic Product rule, fails validation because its WooCommerce mappings are empty, and emits a parseable but useless empty array. The three SSQ3 findings—missing review, aggregateRating, and offerCount—remain warnings only. No genuine reviews, ratings, WooCommerce price, or offer count exist in the inspected WordPress record, so adding values would be fabrication. No issue should be created merely to silence those warnings.

The theme Product does contain one separate identity defect: manufacturer is an inline Organization named “New Tech Machinery,” while Yoast defines the site Organization as “NewTech Machinery” at #organization. The nested node has no @id, so it is not connected to Yoast’s authoritative entity.

The smallest correct direction is:

1. Retire the homepage custom Schema Pro graph and invalidate its cached output. Keep Yoast authoritative for WebPage, BreadcrumbList, WebSite, and Organization.
2. Stop Schema Pro’s generic Product rule from touching theme-owned flagship machine pages, then link the theme’s manufacturer to Yoast’s Organization identity.
3. Disable Schema Pro’s stale SiteNavigationElement mapping to legacy WordPress menu 42; the visible site navigation is hardcoded in the theme and is already the authoritative source.

Recommended issue count: **3**. There should be no issue for the SSQ3 optional warnings and no issue to add an item URL to Yoast’s final homepage crumb.

The local homepage WordPress post is ID 199. Its wp_schema_pro_optimized_structured_data post meta contains the complete custom graph. Schema Pro’s renderer at wp-content/plugins/wp-schema-pro/classes/class-bsf-aiosrs-pro-markup.php:353 returns that cache before it reevaluates the current rule, so changing or drafting rule 18578 without invalidating the cache can leave the defect live.

The local SSQ3 WordPress product is ID 18601. It has no WooCommerce price, regular price, or sale price; rating count and review count are 0; average rating is 0; reviews are disabled. Schema Pro’s mapped Product rule therefore has no real offer or review data to emit. Schema Pro’s template validation at classes/class-bsf-aiosrs-pro-schema-template.php:640 rejects the mapped record and leaves the empty JSON array.

### Existing DB remediation prior art

Two existing replayable scripts already own part of the mechanism needed for a future fix:

- scripts/db/041-unpublish-placeholder-custom-schema.sh drafts only “Sitewide Custom Schema” when placeholder guards match.
- scripts/db/042-purge-placeholder-schema-cache.sh deletes only cache rows containing NEWTECH-UPDATE.

Script 041 explicitly inspected “Home Page Custom Schema” and deliberately left it active because it was populated and contained no placeholder strings. This investigation reverses that **scope decision**, not the safe implementation pattern: the graph is real JSON, but current production proves its entity modeling is invalid, duplicated, and partly hidden/stale. Any implementation issue must say that plainly, use the next migration number (050), resolve records by stable title rather than environment-specific ID, guard on a unique payload signature, and follow the documented direct docker exec/wp eval-file runner workaround. Script 042 must not simply be widened: its NEWTECH-UPDATE needle is correctly scoped to a different defect.

### Global Schema Pro settings

- schema-location: head
- schema-validation: enabled
- yoast-compatibility: enabled
- Global site navigation: WordPress menu ID 42
- Global sitelink search box: enabled
- Global breadcrumb: enabled
- Organization source: general Schema Pro organization settings, with Yoast compatibility active

Schema Pro’s Yoast compatibility prevents its ordinary global Organization, WebSite/search, and Breadcrumb schemas from being emitted. It does **not** inspect and reconcile arbitrary entities inside custom post 18578. That is why the custom graph still duplicates Yoast’s WebPage and BreadcrumbList.

The global SiteNavigationElement is built by classes/schema/global/class-bsf-aiosrs-pro-schema-global-site-navigation-element.php:33 from menu ID 42. Production script 3 matches that menu’s item set. The visible desktop and mobile navigation instead come from app/inc/desktop-nav.php and app/inc/mobile-nav.php, so the schema mapping is attached to a legacy, non-authoritative menu.

### Confidence boundary

The production database was not queried directly. Record IDs and option values are confirmed in the local DevKinsta database; the production payload is confirmed by raw HTML. The attribution of production script 2 to record 18578 is high confidence because the stored custom JSON and production block match after the single expected hostname replacement, and the block is wrapped in Schema Pro’s own optimization comments. A production-side read of the rule and homepage cache is still required before implementation to prove whether production is rendering the live rule or a stale cached copy.

Schema Pro’s official documentation confirms that custom JSON-LD is stored as Custom Markup and applied through target-location rules: [Schema Pro custom schema markup](https://wpschema.com/docs/add-custom-schema-markup/).

Wistia’s public media metadata for kdv2kphni1 reports createdAt 2024-04-09T13:18:28Z. That is evidence for the media record’s creation time, not proof of the first public upload/publication date required by Google.

**Best fix:** Remove the VideoObject with the custom graph. If video rich-result eligibility is intentionally restored later, content should supply and verify the first publication date and representative Wistia thumbnail; code/configuration should retain embedUrl and remove the false homepage contentUrl unless a real media-file URL is available.

### 7.5 Additional confirmed homepage quality issue

Post 18578 includes a six-question FAQPage. Each question string appears only once in the raw homepage HTML—inside JSON-LD—so the FAQ content is not visible to users. It also contains support@stg-newtech-themeupdate.kinsta.cloud. Google’s general structured-data guidelines require markup to represent visible page content and favor fewer accurate properties over extra inaccurate markup: [Google structured-data policies](https://developers.google.com/search/docs/appearance/structured-data/sd-policies).

This was not one of the 14 reported errors, but it is another reason to retire the custom graph rather than patch its failing fields one by one.

### Additional SSQ3 Organization identity conflict — confirmed

The theme’s manufacturer fragment is:

~~~json
"manufacturer": {
  "@type": "Organization",
  "name": "New Tech Machinery"
}
~~~

Yoast independently defines https://newtechmachinery.com/#organization with the name “NewTech Machinery.” The inline manufacturer has no @id, so a graph consumer cannot know that the two nodes are intended to represent the same company. This is not one of the three Google Product warnings, but it is a confirmed disconnected-entity conflict. app/inc/machine-schema.php:64 owns the inline node; the theme should reference the canonical organization @id and use one authoritative name.

One semantic question remains: Google says AggregateOffer describes multiple offers and should not be used merely for product variants. The data file describes $85,000 as the base machine and $143,000 as a top configured build. Product/SEO owners must confirm that these are genuine offers rather than a configuration range. That question is separate from the three warnings and is not grounds to invent offerCount.
